Client-Side Verification

Checking Version Declaration in pom.xml

Description:
Version pinning is recommended as the baseline mitigation and is supported by most clients to protect against supply chain attacks. Specifying precise versions for packages and transitive dependencies, rather than an open range, will mitigate forced upgrade or downgrade attacks Eg. “[3.5.4]” rather than “[3.5)” or “(3.5.4)”. Also the use of Stable and Final release of packages is recommended. Use of Alpha, Beta, Release, Milestone, RC, CR, Snapshot versions should be avoided.

Summary:

Recommendation:

  1. Using a single exact version number within closed range [ ], like [1.2.1] is recommended.
  2. Stable or Final release version should be used. Any intermediate versions should be avoided.

Unused Packages

Description:
Unused Dependencies can be a major overhead for any project, they can affect performance of your application. Using extra dependencies also provides an Expanded Attack Surface to attackers, as unused dependencies are generally ignored and not kept up to date thus create chances of vulnerabilities. Removing Unused Dependencies makes code clean and reduces complexity. If many unused dependencies are present, then it can be hard for other developers to figure out which of them are being used and which are not. They can also create issues while upgrading your project to newer versions.
